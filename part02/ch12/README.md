# 구성 중앙화

마이크로서비스에서 사용하는 구성 설정을 중앙화 해서 응집성을 높일 수 있다.

## 스프링 클라우드 컨피그 서버로 구성 중앙화 하기

컨피그 서버는 다른 마이크로서비스와 마찬가지로 에지 서버의 뒤에 위치하게 된다. 432 그림 참고 

```
* 구성 중앙화 유형

- 깃 저장소
- 로컬 파일 시스템 (이번 단원에서 사용)
- 하시코프 볼트
- JDBC 데이터베이스
```
```
* 구성 서버 의존성

implementation 'org.springframework.boot:spring-boot-starter-actuator'
implementation 'org.springframework.boot:spring-boot-starter-security'
implementation 'org.springframework.cloud:spring-cloud-config-server'
```
### 구성 중앙화 메커니즘 선택하기
```
클라이언트가 먼저 구성을 결정하고 유레카 서버에 등록할 수 있다 이 방식은 유레카의 구성 정보를 구성 서버에 저장할 수 있다.
다른 방식으로 클라이언트를 유레카에 먼저 등록하고 구성 정보를 가져오게 하는 방법이 있다.

다른 방식에 대한 내용은 spring cloud docs 을 참고 (이번 단원에서는 첫번째 방법을 사용한다.)

구성을 먼저 가져오면 구성 서버가 단일 장애점(SPOF, 구성을 먼저 가져올 때 오류가 발생하면 다음 작업으로 넘어가지 못하는 문제)
이 될 수 있다. 이 문제는 쿠버네티스를 사용해서 해결한다. 433p (쿠버네티스는 part 3 에서 설명)
```
### 구성 서버 보안 및 API
```
구성 서버는 게이트웨이 뒤에 위치하며 외부로 노출되지 않는다. 기본적으로 게이트웨이 접근 요청은 https 를
사용해서 도청을 방지하며 마이크로 서비스 내에서는 시큐리티 http 기본 인증으로 클라이언트를 식별할 수 있다.

구성서버 또한 스프링 시큐리티 기본 인증을 사용하며 환경 변수에서 값을 설정한다 env, docker-compose 참고
구성 저장은 로컬 파일 시스템을 사용한다 (/config-repo)

구성 저장소는 접근 권한이 있는 사람이 암호 등의 민감한 정보를 훔쳐가는 상황을 피하고자 구성 서버에서 정보를
암호화해서 디스크 저장한다.

구성 서버는 대칭키와 비대칭키를 지원하는데 여기서는 대칭키를 사용한다. (비대칭 키는 좀 더 복잡하다.)

비대칭 키에 대한 내용은 아래를 참고
cloud.spring.io/spring-cloud-static/spring-cloud-config... key_management 
```
```
* 클라이언트가 검색할 수 있는 구성 서버 REST API

/actuator: 액추에이터는 필요한 정보만 노출해야한다.
/encrypt 및 /decrypt: 중요한 정보를 암호화하고 해독하기 위한 엔드포인트 상용 환경에서는 잠금한다.
/{microservice}/{profile}: 지정한 마이크로서비스의 스프링 프로필 구성 반환
```

### 구성 설정 중앙화 하기
```
지금까지 구현한 구성 정보를 로컬 저장소인 config-repo 파일로 옮긴다. 공통적으로 사용되는 정보는
application.yml 파일로 만들고 각 서비스별 구성 정보를 구현한다. 

구성 정보를 추출한 마이크로서비스에는 구성 서버를 통해서 구성 저장소의 정보를 가져올 수 있도록
아래 의존성을 추가하고 구성서버 연결 설정 yml 파일을 구현한다. 

// 클라우드 컨피그 의존성
implementation 'org.springframework.cloud:spring-cloud-starter-config'
implementation 'org.springframework.retry:spring-retry'

// 클라우드 오류 발생 해결 추가
implementation('org.springframework.cloud:spring-cloud-starter-bootstrap')

config-rep, 각 마이크로 서비스의 application.yml 참고
```

## 클라우드 컨피그 서버 사용해보기
```
* 구성 서버 API 로 구성 조회

curl https://dev-usr:dev-pwd@localhost:8443/config/product/docker -ks | jq

구성은 우선순위에 따라서 반환된다. server.port 8080 이 server.port 7001(복합 상품) 보다 먼저
반환됐으므로 8080 포트를 사용하는 것이다.

민감한 정보인 유레카와 래빗의 암호는 디코딩 한 값이 반환된다.
```
```
* 민감한 정보의 암호화 및 해독 

curl -k https://dev-usr:dev-pwd@localhost:8443/config/encrypt --data-urlencode "hello world"

curl 을 사용해서 /encrypt 엔드포인트 호출시 --data-urlencode 플래그를 사용해서 + 등의 특수 문자를
제대로 처리해야 한다.

curl -k https://dev-usr:dev-pwd@localhost:8443/config/decrypt -d 암호화된 값
위 커맨드로 암호화된 값을 해독할 수 있다.

암호화된 값을 환경 변수 .env 에서 사용하려면 앞에 {cipher} 접두사를 붙여야 한다. .env 참고 
```
#### + 트러블 슈팅(스프링 클라우드)

구성 중앙화를 구현하면서 기존 코드를 대규모로 수정하니 게이트웨이에서 도커 프로필을 사용하지 못하는 문제가 발생했다.

복잡하고 많은 양의 코드를 고칠 때 일어나는 공통적인 문제로써 문제 원인을 찾을 때 우선적으로 고려할 지침을 정리한다.
```
1. 오타 체크

- 가장 빈번하고 지금까지 가장 많이 마주친 문제이다. 컴파일시 오류를 잡아줄 수 없는 영역에서의 오타가 문제를
일으킨 경우가 많았다. 대규모로 수정 해야 한다면 짧게 끊어가면서 리팩토리 하고 착지점을 찾는 것을 추천한다.

2. 코드 논리 체크

- 게이트웨이는 시큐리티 인증을 사용해서 접근을 관리한다 구성 서버를 추가하면서 게이트웨이에서 시큐리티
접근을 허용하지 않으면 구성 서버에 접근 할 수 없다.

새롭게 추가할 모듈이나 수정해야하는 코드가 있다면 가장 최근에 구현한 모듈부터 먼저 체크하고 리팩토링
할 때 필요한 것들을 먼저 확인 하는 것을 추천한다.

3. 의존성 문제

- 필요한 의존성이 없어서 애플리케이션 구동시 발생하는 문제다. 이 문제는 1 2 와는 비교적 수월하게 리팩토링이
모두 이루어지고 나서 필요한 의존성만 추가해줄 수도 있다.

이 문제는 보통 구글링 또는 공식 문서를 통해서 해결해야 하는데 구글링으로 간단하게 찾을 수 없는 문제는 공식 문서를
통해서 해결하는 것을 추천한다.
(보통 검색으로 해결할 수 있는 문제임)

1 2 3 을 하나로 묶어서 하나의 작업 단위로 나누어 해결하면 대량의 코드를 수정하기 전 발생하는 문제를 미리
해결할 수 있다. 모든 코드를 수정하고 문제를 찾는 것은 비효율적이다.

4. 버전 문제 (중요)

- 의존성 버전이 맞지 않아서 발생하는 문제가 있다. 스프링 클라우드를 사용하는 모든 모듈에서 버전이 맞지 않는
하나의 모듈이 문제를 일으킨다던지 개별 모듈의 개별 라이브러리에서 버전 문제가 발생하는 경우

공통적으로 사용하는 라이브러리가 있다면(ex 스프링 클라우드) 설계단계에서 모두 같은 버전을 사용해야 한다.
그외 공통적으로 사용되는 것들도 모두 같은 버전을 사용하는 것을 권장한다. 
```
